  <name>Task 1: Chapter-aware token-based chunking</name>
  <files>rag/chunker.py, requirements.txt</files>
  <action>
Rewrite `rag/chunker.py` to implement chapter-aware token-based chunking:

1. Add `build_chapter_map(documents)` function that scans loaded PDF pages to build a `page_number -> chapter_name` mapping:
   - Use regex pattern `r"Chapter\s+\d+[\s\-\u2013:]+(.+)"` (case-insensitive) on the first 500 chars of each page
   - Track `current_chapter` as pages are iterated — pages without a heading inherit the previous chapter
   - Default to "Unknown Chapter" for pages before the first chapter heading
   - Return `dict[int, str]` mapping page numbers to chapter names like "Chapter 12: Friction"

2. Replace `RecursiveCharacterTextSplitter(chunk_size=..., chunk_overlap=...)` with `RecursiveCharacterTextSplitter.from_tiktoken_encoder(model_name="gpt-4o-mini", chunk_size=800, chunk_overlap=100, separators=["\n\n", "\n", ". ", " ", ""])`.

3. Update `chunk_documents(documents, chunk_size=800, chunk_overlap=100)` signature:
   - First call `build_chapter_map(documents)` to get the page-to-chapter mapping
   - Group documents by chapter using the chapter map — create separate document lists per chapter
   - Split each chapter's documents independently (so chunks never cross chapter boundaries per user decision)
   - After splitting, attach `chapter` metadata key to each chunk based on the page-to-chapter mapping
   - Remove env var fallback for CHUNK_SIZE/CHUNK_OVERLAP — use parameters directly (800/100 defaults)

4. Add `langchain-classic>=1.0.3` to `requirements.txt`. Update `langchain>=0.2.0` to `langchain>=1.2.0` and `langchain-core>=0.2.0` to `langchain-core>=1.2.0` and `langchain-chroma>=0.1.0` to `langchain-chroma>=1.1.0` to match installed versions.

Tables and figure captions must stay in the same chunk as surrounding text — the separators list already handles this since they only split on whitespace/sentence boundaries, not on table markers.

<task type="auto">
  <name>Task 2: Cosine-distance vector store and score-threshold retriever</name>
  <files>rag/vector_store.py, rag/retriever.py</files>
  <action>
Update `rag/vector_store.py`:

1. In `build_vector_store()`, add `collection_configuration={"hnsw": {"space": "cosine"}}` to the `Chroma.from_documents()` call. This is REQUIRED for `similarity_score_threshold` search type to work — without it, Chroma raises ValueError because it cannot determine the distance metric.

2. In `load_vector_store()`, add `collection_metadata={"hnsw:space": "cosine"}` to the `Chroma()` constructor call so the loaded store knows its distance metric. (Note: `Chroma()` constructor uses `collection_metadata` not `collection_configuration` — only `from_documents` uses `collection_configuration`.)

Update `rag/retriever.py`:

1. Change `get_retriever()` to use score thresholding:
   - Change `search_type` from `"similarity"` to `"similarity_score_threshold"`
   - Change `search_kwargs` to `{"k": 6, "score_threshold": 0.35}`
   - Update function signature: `get_retriever(vector_store, top_k=6, score_threshold=0.35)`
   - Remove env var fallback for TOP_K — use parameter default directly

2. Update `retrieve_chunks()` similarly:
   - Use `vector_store.similarity_search_with_relevance_scores(query, k=top_k)` instead of `similarity_search`
   - Filter results where score >= score_threshold
   - Return only the Document objects (not scores) for backward compatibility
   - Update default top_k to 6

Starting threshold is 0.35 (Claude's discretion per CONTEXT.md). This can be calibrated later in Phase 3 benchmarking.
  </action>
